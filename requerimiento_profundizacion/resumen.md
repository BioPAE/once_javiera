📋 Resumen: Ingeniería de Requerimientos

La ingeniería de requerimientos es la disciplina encargada de identificar, analizar, documentar y validar las necesidades de los usuarios y del negocio para desarrollar un sistema que realmente resuelva un problema.

🧠 ¿Qué es un requerimiento?

Un requerimiento es una característica, funcionalidad o condición que un sistema debe cumplir. Para que sea de calidad debe ser claro, verificable, consistente, completo y trazable.

Además, los requerimientos pueden encontrarse en distintos niveles:

Necesidad del usuario: expresa el problema o necesidad.
Requerimiento del sistema: define lo que debe hacer el sistema.
Especificación técnica: describe cómo será implementado.
🔧 Requerimientos Funcionales (RF)

Describen qué hace el sistema. Representan funcionalidades, procesos o servicios que el usuario espera.

Ejemplos:

El sistema deberá permitir iniciar sesión.
El sistema deberá mostrar el menú del día.
El sistema deberá generar reportes mensuales.
⚙️ Requerimientos No Funcionales (RNF)

Describen cómo debe comportarse el sistema y qué nivel de calidad debe ofrecer.

Categorías comunes:

Rendimiento (tiempo de respuesta).
Seguridad (protección de datos).
Usabilidad (facilidad de uso).
Confiabilidad (disponibilidad y estabilidad).
Compatibilidad y portabilidad.

Ejemplo:

El sistema deberá responder a las consultas en menos de 2 segundos.

💎 Atributos de Calidad (ISO 25010)

La norma ISO 25010 define características que determinan la calidad del software:

Adecuación funcional
Eficiencia de desempeño
Compatibilidad
Usabilidad
Confiabilidad
Seguridad
Mantenibilidad
Portabilidad

Los RNF son la forma concreta y medible de expresar estos atributos.

🧩 Elicitación de Requerimientos

Es el proceso de descubrir necesidades mediante técnicas como:

Entrevistas
Encuestas
Observación
Prototipos

Para ello se trabaja con los stakeholders, es decir, todas las personas interesadas en el sistema, como usuarios, clientes, administradores o entidades reguladoras.

📝 Historias de Usuario

Son una forma ágil de capturar requerimientos desde la perspectiva del usuario.

Formato:

Como [usuario], quiero [acción], para [beneficio].

Ejemplo:

Como estudiante, quiero ver el menú del día desde mi celular para decidir qué almorzar antes de ir a la cafetería.

Cada historia debe incluir criterios de aceptación, normalmente escritos en formato:

Dado...
Cuando...
Entonces...
