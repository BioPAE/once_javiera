🎯 Objetivo

Aprender a documentar requerimientos de forma profesional mediante casos de uso, el estándar IEEE 830 (SRS) y la revisión cruzada.

1️⃣ ¿Por qué documentar?

La documentación:

Sirve como acuerdo entre cliente y equipo.
Guía el desarrollo y las pruebas.
Conserva el conocimiento del proyecto.
Funciona como evidencia legal y para auditorías.

Regla: A mayor riesgo del proyecto, mayor nivel de documentación.

2️⃣ IEEE 830 (SRS)

Es un estándar para crear el Documento de Especificación de Requerimientos (SRS).

Secciones principales:

Introducción
Descripción General
Requisitos Específicos (RF y RNF)
Apéndices

Un buen SRS debe ser: correcto, claro, completo, consistente, verificable y trazable.

3️⃣ Casos de Uso

Describen paso a paso cómo un usuario interactúa con el sistema para lograr un objetivo.

Elementos principales:

Actor
Objetivo
Flujo principal
Excepciones

Diferencia clave:

Requerimiento = qué hace el sistema.
Caso de uso = cómo se realiza la interacción.
4️⃣ Plantilla de Caso de Uso

Incluye:

ID
Nombre
Actor
Precondiciones
Postcondiciones
Flujo principal
Excepciones
Reglas de negocio
Prioridad
5️⃣ Revisión Cruzada

Consiste en revisar documentos para detectar:

Ambigüedades
Contradicciones
Requerimientos faltantes
Requisitos no medibles
